## 1. Общее назначение

Модуль реализует **исходящую интеграцию ТОИР с 1С:ERP** через HTTP-сервисы (REST-подобный обмен JSON). Функции:

1. **Хранение и чтение параметров подключения** к ERP из безопасного хранилища.
2. **Установка `HTTPСоединение`** к публикации ERP.
3. **Выгрузка документов** в ERP: сборка данных документа → JSON → POST/GET на HTTP-сервис `hs/IntegrationTOIR/V1/*` → разбор ответа → обновление статуса выгрузки.
4. **Сопоставление объектов** двух баз через РС `СоответствияОбъектовИнформационныхБаз` (маппинг GUID приёмник ↔ источник).

Основные бизнес-операции:
- `СоздатьОбновитьЗаказНаВнутреннееПотреблениеВЕРП` (стр. 134)
- `СоздатьОбновитьПроизводствоБезЗаказаВЕРП` (стр. 346)
- `СтатьиСписанияПоНоменклатуреИПодразделению` (стр. 266)
- `ПолучитьСтатусСогласованияЗаказаНаВнутреннееПотребление` (стр. 446)
- `ПолучитьСостояниеРезервированияНоменклатуры` (стр. 517)

Судя по подписям в комментариях (`[Деснол] [Гусаров Н.Н.]`, `//Скрипко А.В.`), код писали разные разработчики в разное время — отсюда стилевой разнобой и дублирование.

**Структура областей:**
- `ПрограммныйИнтерфейс` — экспортное API (соединение, поиск ссылок по УИД/ГУИД).
- `СлужебныйПрограммныйИнтерфейс` — процедуры выгрузки документов и получения данных из ERP.
- `СлужебныеПроцедурыИФункции` — формирование данных запросами, JSON-хелперы, запись в РС маппинга.

### Ключевые поля подключения
- `Сервер` — адрес хоста ERP.

- `Пользователь` / `Пароль` — HTTP-учётка.

### Терминология обмена
- **Приемник** — база, куда выгружали (ERP).
- **Источник** — текущая база (ТОИР).

### Переменные основных процедур
- `АдресРесурса` — путь HTTP-сервиса; `Заголовки` — `Content-Type: application/json`; `HTTPЗапрос`, `ОтветHTTP`.
- `СтруктураДляОтправки` → `СтрокаJSON` — тело запроса.
- `Результат` — `{КодСостояния, ОтветСтрокой, ДанныеJSON}`.
- `data`, `errors` — секции JSON-ответа ERP (латиницей — как в контракте API).
- `Параметры` — структура для `ум_СтатусыВыгрузкиЕРП.ОбновитьСтатусВыгрузки(Ссылка, Параметры, ЕстьОшибка)`; 3-й аргумент `Истина` = зафиксировать ошибку.

## 4. Проблемы с кодом

### 🔴 Критические (баги)

**4.1. `HTTPСоединениеСЕРП`: обращение к несуществующим полям в блоке `Исключение`** — стр. 41
```bsl
Соединение = Неопределено;
Попытка
    Соединение = Новый HTTPСоединение(...);   // если упало — Соединение осталось Неопределено
Исключение
    ТекстСообщения = СтрШаблон("... Srvr=%1;Ref=%2 ...", Соединение.Сервер, Соединение.База, ...);
```
При исключении `Соединение = Неопределено` → `Соединение.Сервер` бросит **новое исключение прямо в обработчике ошибки**, затирая исходную. Плюс у `HTTPСоединение` **нет свойства `.База`**. Значения нужно брать из `СтруктураДляПодключения`.

**4.2. Ложная диагностика через `ОписаниеОшибки()` после `Возврат Неопределено`** — стр. 140, 352, 451, 523
```bsl
Если HTTPСоединение = Неопределено Тогда
    ТекстОшибки = СтрШаблон("Ошибка соединения с ERP: %1", ОписаниеОшибки());
```
Исключение уже перехвачено *внутри* `HTTPСоединениеСЕРП`, здесь `ОписаниеОшибки()` вернёт пустоту/мусор из более раннего сбоя. Реальная причина сбоя не доходит до пользователя. **Разбор и исправление — см. раздел «Исправление 4.2» ниже.**

**4.3. Переопределение переменной `Результат` ломает контракт** — стр. 507-511
```bsl
Если Результат.КодСостояния = 200 Тогда 
    Результат = ДанныеJSON.data.СтатусСогласования;  // была структура — стала строка
КонецЕсли;
Возврат Результат
```
При коде **не 200** функция вернёт **структуру**, при 200 — **строку**. Разный тип у вызывающего. Плюс если `data`/`СтатусСогласования` нет — исключение.

**4.4. `ПолучитьДанныеJSON` вызывается повторно без защиты** — стр. 578, 305, 508
При коде 200 идёт повторный `ПолучитьДанныеJSON(...)` **без `Попытка`**. Если тело — не JSON (например, HTML-страница веб-сервера при 200), будет необработанное исключение.

**4.5. `errors`/`data` читаются из «сырого» `ДанныеJSON`, а не из проверенного `Результат.ДанныеJSON`** — стр. 192, 202, 404, 414
Проверяют `Результат.ДанныеJSON.Свойство(...)`, а значение берут из переменной `ДанныеJSON`. Работает лишь потому, что это одна ссылка — хрупко и путано.

**4.6. Опечатка в парном маркере комментария** — стр. 241
`//Скрипко А.В.{` — открывающая скобка вместо закрывающей `}`.

### 🟠 Существенные

- **4.7.** Нет явного закрытия `HTTPСоединение` (риск при больших циклах).
- **4.8.** Массовое дублирование кода: блок «соединение → заголовки → запрос → JSON → отправка → разбор → статус» скопирован 1:1 в 4 процедурах (134, 346, 446, 517). Исправлять надо в 4 местах (что и видно: в 517 логирование уже переделали на журнал регистрации, в остальных осталось `СообщитьПользователю`).
- **4.9.** При успехе (код 200) `Комментарий` принудительно `""` (стр. 204-205), хотя выше в `errors` могли накопиться предупреждения (191-198) — частичные ошибки при 200 теряются.
- **4.10.** `ум_ПотребностьДляОДиАСУТПиИТ` в запросе выбирается напрямую (стр. 613), без обёртки `ПРЕДСТАВЛЕНИЕ(УНИКАЛЬНЫЙИДЕНТИФИКАТОР(...))` — если реквизит ссылочный, в JSON уйдёт представление, а не GUID. Проверить намеренность.
- **4.11.** Запись документа с проведением внутри интеграции без `Попытка` (стр. 238-239): если проведение упадёт, ERP уже создал документ → рассинхрон.
- **4.12.** `УзелИнформационнойБазыЕРП` с захардкоженным GUID (90) — при пересоздании узла/переносе базы сломается молча.

### 🟡 Мелкие / стиль

- Опечатки: `Подраздедление` (266, 667), `не найденно`, `не одной строки` (233, 235).
- Отсутствие `;` после `Возврат` (79, 85, 122, 276, 342, 511, 580, 702).
- `errors`, `data`, `error` — латиница вперемешку с кириллицей.
- Мёртвый закомментированный код (490-505, 536).

## 5. Edge-cases, ведущие к ошибкам

1. **ERP недоступен / упал HTTP** → п. 4.1 (`Соединение.База`) + п. 4.2 (пустой `ОписаниеОшибки`). Невнятная или неверная ошибка пользователю.
2. **HTTP 200, но тело — не JSON** → падение в 305, 508, 578 (повторный парсинг без `Попытка`).
3. **Ответ 200 без `data`** → `ДанныеJSON.data` (202/414/508) бросит исключение (нет проверки `Свойство("data")`).
4. **`data.Товары[].НоменклатураУИД` пустой/битый GUID** → `Новый УникальныйИдентификатор(...)` (227) бросит исключение прямо в цикле, прервав обработку всех строк.
5. **Несколько строк с одинаковой номенклатурой и КодСтроки** → ветка «Иначе» (234-235) только сообщает, КАП не заполняется — тихая потеря данных.
6. **`ПолучитьСсылкуПоУникальномуИдентификатору`**: если записей несколько, берётся первая произвольная.
7. **`ПолучитьСсылкуПоГУИД` с несуществующим видом** (`Справочники[ВидСсылки]`, 108) → исключение вместо аккуратной ошибки.
8. **`ПолучитьСсылкуПоГУИД`, объект удалён по GUID** → возвращает `Неопределено` (120-123) — вызывающий должен это ждать.
9. **`СтатусСогласования`**, код ≠ 200 → возвращается структура вместо строки (п. 4.3).
10. **Проведение документа (239) падает** → ERP-документ создан, статус выгрузки не обновлён → дубль при повторе (если сервис не идемпотентен).
11. **Безопасное хранилище пустое** (16) → `Новый HTTPСоединение(Неопределено, ...)` (39) упадёт.

## Исправление 4.2 (детально)

### Суть проблемы
`ОписаниеОшибки()` возвращает текст **последнего необработанного** исключения в текущем контексте. Но исключение при создании `HTTPСоединение` уже **перехвачено внутри** `HTTPСоединениеСЕРП` (стр. 38-43). К моменту проверки `Если HTTPСоединение = Неопределено Тогда` в вызывающей процедуре (стр. 139-140) активного исключения нет → `ОписаниеОшибки()` вернёт пустую строку или мусор. Реальная причина обрыва теряется.

**Ключевой принцип:** `ОписаниеОшибки()` информативен только внутри блока `Исключение`, где произошёл сбой. Оттуда ошибку и надо забирать, а не «дотягиваться» до неё после `Возврат Неопределено`.

### Вариант А (рекомендуется): выходной параметр `ТекстОшибки`

Одновременно закрывает и баг 4.1 (берём поля из `СтруктураДляПодключения`, а не из `Соединение`). Обратно совместимо — параметр со значением по умолчанию.

```bsl
Функция HTTPСоединениеСЕРП(СтруктураДляПодключения, ТекстОшибки = "") Экспорт 
	
	Соединение = Неопределено;   
	
	Попытка	
		Соединение = Новый HTTPСоединение(СтруктураДляПодключения.Сервер,, СтруктураДляПодключения.Пользователь, СтруктураДляПодключения.Пароль);	
	Исключение     
		ТекстОшибки = СтрШаблон("Ошибка установки соединения с базой ERP (Сервер=%1; База=%2): %3",
			СтруктураДляПодключения.Сервер, СтруктураДляПодключения.База, ОписаниеОшибки());
	КонецПопытки;
	
	Возврат Соединение;
	
КонецФункции
```

В вызывающих процедурах (134, 346, 446) заменить:
```bsl
HTTPСоединение = HTTPСоединениеСЕРП(СтруктураДляПодключенияКЕРП);
Если HTTPСоединение = Неопределено Тогда     
    ТекстОшибки = СтрШаблон("Ошибка соединения с ERP: %1", ОписаниеОшибки());   // ← пусто/мусор
    ...
```
на:
```bsl
ТекстОшибки = "";
HTTPСоединение = HTTPСоединениеСЕРП(СтруктураДляПодключенияКЕРП, ТекстОшибки);
Если HTTPСоединение = Неопределено Тогда     
    // ТекстОшибки уже содержит реальную причину сбоя
    ОбщегоНазначения.СообщитьПользователю(ТекстОшибки);   
    Параметры = Новый Структура("Комментарий", ТекстОшибки);
    РегистрыСведений.ум_СтатусыВыгрузкиЕРП.ОбновитьСтатусВыгрузки(СсылкаНаДокумент, Параметры, Истина);
    Возврат;
КонецЕсли;
```

Для процедуры с журналом регистрации (517):
```bsl
ТекстОшибки = "";
HTTPСоединение = HTTPСоединениеСЕРП(СтруктураДляПодключенияКЕРП, ТекстОшибки);
Если HTTPСоединение = Неопределено Тогда
    ЗаписьЖурналаРегистрации(НСтр("ru = 'Ошибка соединения с ERP'"),
        УровеньЖурналаРегистрации.Ошибка, , , ТекстОшибки);
    Возврат Неопределено;
КонецЕсли;
```

### Вариант Б (проще, грубее)
Не глушить исключение в `HTTPСоединениеСЕРП`, а пробрасывать (`ВызватьИсключение` после логирования) и ловить `Попытка` в вызывающей процедуре — тогда `ОписаниеОшибки()` в её блоке `Исключение` валиден. Минус — обёртка `Попытка` в каждом вызове увеличивает дублирование.

**Рекомендация:** вариант А — локальный, обратно совместимый, закрывает 4.2 и 4.1 сразу.

## Итог

Модуль **рабочий, но технически «сырой»**: сильное дублирование (4 почти идентичные процедуры), непоследовательная обработка ошибок (`СообщитьПользователю` vs журнал регистрации), реальные баги в ветках ошибок (41, 140, 507) и опасная запись документа с проведением внутри интеграции (239).

**Первоочередное к исправлению:**
1. Стр. 41 — убрать `Соединение.База`/`.Сервер`, брать из `СтруктураДляПодключения`.
2. Стр. 140/352/451/523 — донести реальный текст ошибки (Исправление 4.2, вариант А).
3. Стр. 507 — не переопределять `Результат`, вернуть согласованный тип.
4. Стр. 305/508/578 — обернуть повторный парсинг JSON в `Попытка` + проверять `Свойство("data")`.
5. Стр. 239 — обернуть запись в `Попытка` и логировать сбой проведения.
