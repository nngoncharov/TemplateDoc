# Анализ общего модуля «умИнтеграцияERP»

**Файл:** `TOIR/CommonModules/умИнтеграцияERP/Ext/Module.bsl`
**Объём:** 797 строк
**Тип:** Общий модуль, серверный (HTTP-интеграция ТОИР ↔ 1С:ERP)
**Конфигурация:** ТОИР КОРП 3.0.10.1
**Дата анализа:** 2026-07-06

---

## 1. Общее назначение

Модуль реализует **REST-интеграцию ТОИР ↔ 1С:ERP** через HTTP-сервис (`hs/IntegrationTOIR/V1/...`). Функции:

1. **Установка соединения** с базой ERP (данные подключения — в безопасном хранилище).
2. **Выгрузка документов** в ERP: «Заказ на внутреннее потребление» и «Производство без заказа» (формирование JSON → POST → разбор ответа → обновление статуса выгрузки).
3. **Получение справочных данных** из ERP: статьи списания, статус согласования ДО, состояние резервирования номенклатуры.
4. **Сопоставление объектов** двух баз через РС `СоответствияОбъектовИнформационныхБаз` (маппинг УИД приёмник ↔ источник).

**Структура областей:**
- `ПрограммныйИнтерфейс` — экспортное API (соединение, поиск ссылок по УИД/ГУИД).
- `СлужебныйПрограммныйИнтерфейс` — процедуры выгрузки документов и получения данных из ERP.
- `СлужебныеПроцедурыИФункции` — формирование данных запросами, JSON-хелперы, запись в РС маппинга.

## 2. Параметры и переменные

### API-функции
| Функция | Назначение |
|---|---|
| `СтруктураДляПодключенияКЕРП()` | Читает из безопасного хранилища `{Сервер, База, Пользователь, Пароль}` под привилегированным режимом. |
| `HTTPСоединениеСЕРП(СтруктураДляПодключения)` | Создаёт `HTTPСоединение` (сервер, порт по умолчанию, логин, пароль). |
| `ПолучитьСсылкуПоУникальномуИдентификатору(УникальныйИдентификаторПриемника, ТипПриемника, ТипИсточника)` | По УИД в ERP находит соответствующую ссылку ТОИР через РС маппинга. |
| `УзелИнформационнойБазыЕРП()` | Возвращает предопределённый узел плана обмена по жёстко зашитому GUID. |
| `ПолучитьСсылкуПоГУИД(ГУИД, ТипСсылки)` | Универсально строит ссылку (Справочник/Документ/ПланВидовХарактеристик) по УИД + строковому типу. |

### Процедуры выгрузки `СоздатьОбновить...ВЕРП(СсылкаНаДокумент)`
| Переменная | Назначение |
|---|---|
| `СсылкаНаДокумент` | Выгружаемый документ. |
| `АдресРесурса` | Путь HTTP-сервиса ERP. |
| `ОтветHTTP` / `ОтветЗапросаСтрокой` / `ДанныеJSON` | Ответ и его разбор. |
| `Параметры` | Структура для `ОбновитьСтатусВыгрузки` (ключ `Комментарий`, опц. `СсылкаНаОбъектЕРП`, `СсылкаНаОбъектДО`). |

### Функции формирования данных
| Функция | Назначение |
|---|---|
| `ДанныеЗаказаНаВнутреннееПотребление` / `ДанныеПроизводствоБезЗаказа` | Пакетный запрос (шапка + ТЧ) → структура для JSON. |
| `ПолучитьСтрокуJSON` / `ПолучитьДанныеJSON` | Сериализация / десериализация JSON. |
| `ДобавитьЗаписьВРегистрСоответствияОбъектовИнформационныхБаз` | Запись маппинга УИД в РС `СоответствияОбъектовИнформационныхБаз`. |

## 4. Проблемы с кодом

### Серьёзные

1. **Баг в `HTTPСоединениеСЕРП`** (стр. 39–42): при исключении в конструкторе `Соединение` остаётся `Неопределено`, но в тексте ошибки обращаются к `Соединение.Сервер` / `Соединение.База` → **вторичное исключение** внутри блока `Исключение`, реальная ошибка теряется. Нужно брать `СтруктураДляПодключения.Сервер/.База`. Причём поля `.База` у `HTTPСоединение` вообще нет.

2. **Массовое дублирование кода** — процедуры `СоздатьОбновитьЗаказНаВнутреннееПотреблениеВЕРП` (134) и `СоздатьОбновитьПроизводствоБезЗаказаВЕРП` (346) идентичны на ~90% (соединение, заголовки, отправка, разбор ответа, обработка `errors`, ветка `200`). Отличаются только адресом ресурса, функцией данных и обработкой `data.Товары`. Просится единая процедура `ВыгрузитьДокумент(...)`.

3. **Вызов через собственное имя модуля** `умИнтеграцияERP.ИмяФункции` (стр. 176, 305, 308…). Внутри самого модуля лишнее и медленнее; можно вызывать напрямую.

4. **`Записать(РежимЗаписиДокумента.Проведение)` в цикле выгрузки** (стр. 239) — при синхронизации КАП документ **перепроводится** без `Попытки`. Если проведение упадёт (блокировки, контроль остатков), исключение уйдёт наверх, а статус выгрузки уже частично обновлён.

5. **Пропущенные точки с запятой после `Возврат`** — стр. 79, 85, 144, 166, 187, 276, 296, 356, 378, 399, 453, 474, 511, 702. Синтаксически допустимо, но нарушает стандарт оформления.

### Средние

6. **Опечатки:**
   - `Подраздедление` (стр. 266, 286, 667, 671) — параметр во всех вхождениях.
   - `"не найденно соответствие"`, `"не найденно не одной строки"` (стр. 233, 235) — тексты пользователю.

7. **`Если ...Количество() Тогда`** (стр. 651, 782) — число как булево (`0`=Ложь). Должно быть `> 0`.

8. **Разнобой обработки ошибок**: часть функций пишет через `ОбщегоНазначения.СообщитьПользователю` (интерактивно), часть — в `ЖурналРегистрации` (стр. 525, 547). Для серверной интеграции `СообщитьПользователю` может не дойти до пользователя (фоновое задание) — ошибки молча теряются.

9. **`ПолучитьСостояниеРезервированияНоменклатуры`** (стр. 517): параметр `СписокЗНВП` **не используется** — в `СтруктураДляОтправки` вместо него подставлен `Новый Массив` (стр. 537, старая строка закомментирована). Фильтр по номерам ЗНВП не работает, всегда шлётся пустой массив — функциональный дефект/недоделка.

10. **`Комментарий` в ветке `200` не учитывает уже собранные `errors`** (стр. 205, 417): при коде 200, но с непустым блоком `errors`, комментарий перезатирается пустой строкой — предупреждения ERP теряются в статусе выгрузки.

## 5. Edge-cases

1. **`ПолучитьДанныеJSON` при коде 200, но пустом/битом теле** — в `СтатьиСписания...` (стр. 305) и `ПолучитьСостояние...` (стр. 578) при коде 200 JSON парсится **без `Попытки`**. Если ERP вернул 200 с не-JSON телом → необработанное исключение.

2. **`ДанныеJSON.data.СтатусСогласования`** (стр. 508) — если в ответе 200 нет `data` или нет `СтатусСогласования`, обращение по несуществующему свойству упадёт. Проверок `Свойство(...)` нет (в отличие от блока выгрузки).

3. **`ПолучитьСсылкуПоУникальномуИдентификатору`** возвращает `УникальныйИдентификаторИсточника` (стр. 85), но по имени/назначению ожидается *ссылка*. По факту это УИД, а не `СправочникСсылка`. Вызов на стр. 308 передаёт результат в `ЗначениеЗаполнено` — для УИД результат может быть неожиданным.

4. **`ПолучитьСсылкуПоГУИД`**: `ПолучитьСсылку` создаёт ссылку **не проверяя существование объекта**; проверка идёт через `ПолучитьОбъект() = Неопределено` (стр. 121). Для помеченного на удаление объекта вернётся сама ссылка — вызывающий код может не ожидать «висящую» ссылку.

5. **`errors` как строки vs объекты** — код предполагает, что элементы `errors` это строки (`Комментарий + ТекстОшибки`, стр. 195). Если ERP вернёт массив объектов ошибок, конкатенация со структурой вызовет ошибку приведения типа.

6. **`УдалитьПоследнийСимволВСтроке(Комментарий, 1)`** (стр. 197) — вызывается под `Если Свойство("errors")`; при пустом массиве `errors` цикл не выполнится, комментарий останется пустым, обрезка пустой строки безопасна, но логика хрупкая.

7. **Многострочный документ + `НайтиСтроки` по `Номенклатура+КодСтроки`** (стр. 229) — обрабатывается только случай ровно `= 1`; при дублях (`Иначе`, стр. 234–235) КАП **не проставляется вообще**, хотя дубль может быть легитимным — данные молча теряются.

## Итог

Модуль рабочий, но с серьёзным техдолгом:
- две почти одинаковые процедуры выгрузки (кандидаты на объединение в `ВыгрузитьДокумент(...)`);
- баг в обработчике ошибки соединения (стр. 41, обращение к `Соединение.Сервер/.База` при `Соединение = Неопределено`);
- недоделанная фильтрация в `ПолучитьСостояниеРезервирования` (стр. 537 — всегда пустой массив);
- жёстко зашитый GUID узла плана обмена (стр. 90);
- разнобой в обработке ошибок (интерактив vs журнал регистрации).

### Рекомендуемые направления доработки
1. Починить `HTTPСоединениеСЕРП`: в тексте ошибки использовать `СтруктураДляПодключения.Сервер`, убрать несуществующее `.База`.
2. Вынести общую логику выгрузки документа в единую процедуру, оставив специфику (адрес, функция данных, постобработка `Товары`) параметрами.
3. Обернуть парсинг JSON при коде 200 в `Попытку` (статьи списания, статус согласования).
4. Восстановить использование `СписокЗНВП` в `ПолучитьСостояниеРезервированияНоменклатуры`.
5. Вынести GUID узла и код `200` в константы/предопределённые значения; унифицировать обработку ошибок через журнал регистрации.
