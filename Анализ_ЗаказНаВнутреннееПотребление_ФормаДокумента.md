# Анализ модуля формы документа «Заказ на внутреннее потребление»

**Файл:** `TOIR/Documents/ЗаказНаВнутреннееПотребление/Forms/ФормаДокумента/Ext/Form/Module.bsl`
**Объём:** 1658 строк
**Конфигурация:** ТОИР КОРП 3.0.10.1 (доработка типовой формы УТ/ERP)
**Дата анализа:** 2026-07-06

---

## 1. Общее назначение

Типовая форма УТ/ERP документа **«Заказ на внутреннее потребление»**, глубоко доработанная под подсистему ТОИР (префикс `торо_`). Функциональные блоки:

1. **Стандартный каркас БСП** — Свойства (доп. реквизиты), ПодключаемыеКоманды, ВерсионированиеОбъектов, УправлениеДоступом, ДатыЗапретаИзменения, ОценкаПроизводительности (замеры ключевых операций).
2. **Работа с ТЧ «Товары»** — подбор номенклатуры, пересчёт упаковок/единиц, характеристики, серии.
3. **Механизм замены номенклатуры на аналоги** (`торо_ЗамененныеАналогамиМатериалы`) — центральный и самый рискованный блок.
4. **Workflow утверждения заказов на ВП** (`УтверждениеЗаказовНаВП`) — статусы документа и строк, комментарии к утверждению, права утверждающего.
5. **Интеграция документов ТОИР** — привязка заказа ЕРП к документам-основаниям ТОИР через РС `торо_ИнтеграцияДокументов`.

## 3. Магические числа и строки

| Значение | Смысл |
|---|---|
| **`ЕстьАналоги = 0`** | Есть доступные аналоги, ещё не применены → при попытке редактирования открывается форма подбора аналогов. |
| **`ЕстьАналоги = 1`** | Аналоги применены (строка участвует в замене). |
| **`ЕстьАналоги = 2`** | Аналогов нет / механизм неприменим → строка редактируется свободно (стр. 488, логика запроса 1150–1156). Флаг-состояние без перечисления — главный источник «магии» в модуле. |
| `ДАТАВРЕМЯ(1,1,1)` (стр. 1166) | «Пустая дата» = аналог без даты завершения (бессрочный). |
| `""""` (стр. 1112) | Пустой `IDРемонта` для строк заказа без привязки к ремонту. |
| `"АвтоТест"` (стр. 20) | Стандартный ранний выход при автотестировании. |
| Строки-имена событий (`"ЗаписанЗаказНаВнутреннееПотреблениеТоваров"`, `"ИзмененСтатусУтвержденияЗаказаНаВП"` и др.) | Каналы `Оповестить` / `ОбработкаОповещения` между формами. |

## 4. Проблемы с кодом

### Серьёзные

1. **`НайденныеСтроки[0]` без проверки на пустоту** — систематический паттерн. Стр. 1212 `НайденныеСтроки[0].ЕстьАналоги = ...`: если запрос вернул `КодСтроки`, которого уже нет в ТЧ (рассинхрон), будет **индекс за границей**. То же в стр. 1338, 1379, 1425 — обращение к `[0]` без проверки `Количество() > 0`.

2. **`Если ИДБезОснований.Количество() Тогда`** (стр. 982) — число используется как булево. Работает (`0` = Ложь), но неявное приведение, плохой стиль; должно быть `> 0`.

3. **`ЗаполнитьЗначенияСвойств(НоваяСтрокаТЧ_ЗаказыДоЗамены, Объект)`** (стр. 1277, 1336) — копирование свойств из `Объект` (документа) в строку ТЗ по совпадению имён. Хрупко: любое переименование реквизита (`Подразделение`, `Ответственный`, `Склад`) молча ломает заполнение.

4. **Дублирующийся блок `Если Упаковка...Иначе...ЕдиницаИзмерения`** — повторяется 6+ раз почти идентично (стр. 1282–1286, 1298–1302, 1342–1346, 1358–1362, 1384–1388). Просится вынести в функцию `ЕдиницаИзмеренияПоУпаковке(Упаковка, Номенклатура)`.
   **Явный copy-paste баг:** в стр. 1298–1302 блок повторно присваивает `НоваяСтрокаТЧ_ЗаказыДоЗамены.ЕдиницаИзмерения` (уже присвоено в стр. 1283) — второй раз должна была заполняться строка `РезультатЗамены`, а не `ЗаказыДоЗамены`.

### Средние

5. **Логическая мешанина в `ПередЗаписью`** (стр. 150–155) — условие с `ИЛИ`/`И` без явных скобок вокруг всей группы. Приоритет работает, но читаемость нулевая, легко ошибиться при правке.

6. **Опечатки:**
   - `ПолучитьМассивТиповНоменклаутры` (стр. 753) — «Номенклаутры».
   - `"обязатален ввод комментария"` (стр. 1529) — опечатка в тексте пользователю.

7. **Дублирование `ВсеСтрокиОтменены = Истина`** (стр. 1576 и 1579) — переменная инициализируется дважды подряд. Мёртвая строка (вероятно, при копировании потеряна инициализация другой переменной).

8. **`ЭтаФорма.ТолькоПросмотр = Объект.торо_СозданПоПотребности`** (стр. 74) — вся форма → «только просмотр» при создании по потребности. Потенциальный конфликт с workflow-логикой утверждения (два механизма блокировки).

9. **Смешение стилей чтения перечислений** — на клиенте `ПредопределенноеЗначение("Перечисление...")`, на сервере `Перечисления....`. Десятки строковых путей — при опечатке в строке ошибка выявится только в рантайме.

10. **`ОбработатьЗаменуНоменклатурыНаАналоги`** (стр. 1403) — `КодСтрокиНоменклатуры` инициализируется `0`, передаётся по ссылке в `ОбработатьЗаменуНаАналогВСтроке` в цикле, но между итерациями явно не переустанавливается — скрытая зависимость от внешней процедуры.

## 5. Edge-cases

1. **Рассинхрон `КодСтроки` между ТЧ и результатом запроса** — если запрос `ОтметитьНаличиеАналогов` вернул код удалённой параллельно строки, `НайденныеСтроки[0]` (стр. 1212) упадёт. Актуально при быстрых интерактивных правках.

2. **`МаксимальныйКодСтроки` не восстанавливается** — код инкрементится (стр. 485, 916), но при удалении строк не убывает. При копировании документа или ручной правке реквизита возможны коллизии кодов, ломающие привязку аналогов (держится на `КодСтроки`).

3. **Запись документа внутри `ПередЗаписью`** — при `ОбязателенВводКомментария()` ставится `Отказ = Истина`, вызывается `ПоказатьВводСтроки`, а в колбэке `торо_КомментарийЗавершениеВвода` — `ЭтотОбъект.Записать(...)` (стр. 1539). Повторный вход в `ПередЗаписью`. Если требование комментария не снялось — возможен **цикл** «запись → требуем комментарий → запись».

4. **`НайденныеСтрокиТоваров[0]` в ветке «нет строки-аналога»** (стр. 1338) — если строку в ТЧ «Товары» по `КодСтроки` не нашли (удалили, а аналог остался), обращение к `[0]` падает без диагностики.

5. **Пустой `Объект.Ссылка` (новый документ) в запросах аналогов** — `ОтметитьНаличиеАналогов` (стр. 1201) передаёт пустую ссылку; соединение по `ЗаказНаВП` (стр. 1082) не найдёт ремонтов → все строки уйдут в ветку «заказы без ремонтов» (стр. 1123). До записи документа аналоги по ремонтам не подтягиваются — вероятно, ожидаемо, но неочевидно.

6. **`ПодразделениеПриИзменении` не проверяет заполнение склада** — вызывает `ЗаполнитьСклад`; если склад был выбран вручную, он молча перезапишется.

7. **Пустой/неуникальный `Ремонт.Представление`** — `СделатьЗаписиВРСИнтеграцияДокументов` (стр. 966) использует `Ремонт.Представление` как `Запись.ID` (стр. 976) — привязка к строковому представлению, а не к значению. Если представление пустое/неуникальное, записи РС перезатрут друг друга.

## Итог

Самый рискованный участок — **блок замены на аналоги (стр. 1237–1434)**:
- много обращений к `[0]` без проверок;
- дублирование кода с одним явным copy-paste багом (стр. 1298–1302);
- вся конструкция держится на числовом магическом флаге `ЕстьАналоги (0/1/2)` и на `КодСтроки`, целостность которого ничем не гарантируется.

### Рекомендуемые направления доработки
1. Рефакторинг блока аналогов: вынести дублирование `ЕдиницаИзмерения` в функцию, добавить проверки `Количество() > 0` перед `[0]`.
2. Заменить магический флаг `ЕстьАналоги (0/1/2)` на перечисление либо описать константами с говорящими именами.
3. Гарантировать уникальность/целостность `КодСтроки` и `МаксимальныйКодСтроки` при удалении/копировании строк.
4. Устранить риск рекурсивной записи в `ПередЗаписью` / `торо_КомментарийЗавершениеВвода`.
