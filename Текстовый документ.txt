1. Анализ запроса из Денвика (экстрактор)
Пакетный запрос из 14 секций: собирает основания ремонтов (выявленные дефекты, внешние основания, внеплановые обходы, план-график, плановые мероприятия), соединяет их с общими данными по ремонтам, плановыми/фактическими датами, отменами и актами, и на выходе даёт понедельную сводку по подразделениям: План / Факт / Отменено / СреднееЧас / СреднееДней.
Параметры экстрактора: &ДатаНачала, &ДатаОкончания, &ИспользоватьОтборПоПодразделениям, &Подразделение, &БратьТолькоПлановыеЗвершенныеРаботы.
2. Анализ запроса из отчёта
Тот же запрос в виде строкового литерала 1С (с | и удвоенными кавычками), но с патчем Деснол [51305] [18.03.2026], который виден по препроцессорным меткам #Удаление / #Вставка. Параметры отчёта: &НачалоПериода, &ОкончаниеПериода вместо &ДатаНачала/&ДатаОкончания, остальные совпадают.
3. Что изменилось и какие поля
Изменилось поле Завершен в пакете ВТ_ДанныеПредварительно — это единственное содержательное изменение патча. Было простое торо_ОбщиеДанныеПоРемонтам.Завершен, стало условное выражение:
ВЫБОР
    КОГДА &БратьТолькоПлановыеЗвершенныеРаботы
        ТОГДА ВЫБОР
                КОГДА ТИПЗНАЧЕНИЯ(...ДокументНачалаЦепочки) = ТИП(Документ.торо_ПланГрафикРемонта)
                    ТОГДА торо_ОбщиеДанныеПоРемонтам.Завершен
                ИНАЧЕ ЛОЖЬ
              КОНЕЦ
    ИНАЧЕ торо_ОбщиеДанныеПоРемонтам.Завершен
КОНЕЦ КАК Завершен
Новых выходных полей нет — изменилась логика существующего поля и появился новый параметр &БратьТолькоПлановыеЗвершенныеРаботы (булево). Обратите внимание: в имени параметра опечатка («Звершенные» без «а») — она есть в обоих файлах, менять её нельзя, иначе параметр не совпадёт с кодом, который его устанавливает.
Важный факт, который я обнаружил при диффе: в присланном вами файле экстрактора Денвика этот патч уже вставлен — блок ВЫБОР с новым параметром там уже есть (он даже отличается отступами-табами от остального файла). После разрешения меток #Вставка/#Удаление и переименования параметров периода тексты запросов побайтно идентичны (проверено программно). То есть единственное реальное расхождение — имена параметров периода.
4. Как правильно изменить запрос для Денвика
Я взял запрос из отчёта (с применённой вставкой Деснол), очистил его от строковых обёрток 1С (|, "") и заменил параметры на те, что использует Денвик: &НачалоПериода → &ДатаНачала, &ОкончаниеПериода → &ДатаОкончания. Результат — в файле «Запрос_для_ДЕНВИК_итог.txt», его можно целиком вставить в экстрактор.
Дополнительно в самом экстракторе Денвика нужно зарегистрировать новый параметр БратьТолькоПлановыеЗвершенныеРаботы и задать ему значение (тип Булево). Без этого запрос упадёт с ошибкой «Не задано значение параметра».
Изменение логики, о котором обязан сообщить: значение этого параметра влияет на счётчик «Факт» в итоговой выборке. При ИСТИНА завершёнными будут считаться только ремонты, цепочка которых началась с документа «План-график ремонта»; ремонты по дефектам, внешним основаниям и мероприятиям перестанут попадать в «Факт», даже если фактически завершены. Чтобы Денвик отдавал данные как раньше, установите параметр в ЛОЖЬ — тогда поведение полностью совпадает со старой версией. Если же цель патча 51305 — считать факт только по плановым работам, ставьте ИСТИНА, но цифры «Факт» в выгрузке уменьшатся.
Самопроверка
Выполнена программно: итоговый запрос после нормализации пробелов идентичен запросу из экстрактора Денвика и семантически совпадает с новым запросом отчёта (единственное отличие — имена параметров периода, что на результат не влияет, так как это лишь переименование). Структура пакета корректна: 12 временных таблиц создаются, ВТ_ВД и ВТ_ВО уничтожаются после использования, финальная секция возвращает данные. Противоречий с запросом из отчёта нет.
